Git版本控制一般流程


大致分为4个板块：

  • 工作目录：存放正在写的代码（当我们新版本开发完成之后，就可以进行新版本的提交）
  • 暂存区：暂时保存待提交的内容（新版本提交后会存放到本地仓库）
  • 本地仓库：位于我们电脑上的一个版本控制仓库（存放的就是当前项目各个版本代码的增删信息）
  • 远程仓库：位于服务器上的版本控制仓库（服务器上的版本信息可以由本地仓库推送上去，也可以从服务器抓取到本地仓库）

它是一个分布式的控制系统，因此一般情况下我们每个人的电脑上都有一个本地仓库，由大家共同向远程仓库去推送版本迭代信息。

通过这一系列操作，我们就可以实现每开发完一个版本或是一个功能，就提交一次新版本，这样，我们就可以很好地控制项目的版本迭代，想回退到之前的版本随时都可以回退，想查看新版本添加或是删除了什么代码，随时都可以查看。

本地基本操作

初始化和查询状态

git init—————-将当前文件夹作为一个本地仓库，会自动生成一个.git目录
git status—————-查看当前状态（处于哪个分支，哪些文件没被追踪，暂存区内是否还有内容没提交到本地仓库…）

将文件添加到暂存区
新添加的文件是未追踪状态，需要将它添加到暂存区才会被git追踪

git add 文件名
git add . —————-一次性添加目录下所有文件
注：创建.gitignore文件，里面的文件不会被git检查

将暂存区中的内容提交到本地仓库
用commit可以将暂存区里的内容全部提交到git本地仓库
每次提交都有一个commit ID，对应一个版本，HEAD指向当前版本

git commit -m '备注信息'
用-m给这次提交写个备注

文件被改之后，要将这个被改的文件再次add加入暂存区，之后才能commit到本地仓库。也就是说，被commit的文件必须要先放在暂存区里（先add后commit）
在git status中，若modified是红色，说明修改没有被放入暂存区，绿色说明已经被放入暂存区

git commit -a -m '备注信息'—————-可以自动完成add的操作(前提是没有新文件)，省去一条指令

查看提交(版本)记录

git log
git log --oneline -a --graph—————-oneline提供简洁版本，graph可以显示啥时候出现分支与合并
git log --oneline --all --graph—————-推荐用这个，all可以看到远程仓库的情况

查看某次提交的详细内容
红色是这次提交之前的内容，绿色是这次提交之后的内容

git show commit ID
如果直接git show，就是显示最近一次提交

回滚：回退到某个版本

git reset commit ID —————-工作区内容不变，将暂存区内容回退到指定的版本
git reset --hard commit ID —————-将工作区和暂存区都回退到指定版本

回滚后，之后的记录都消失，如果想回去，则用git reflog查看操作记录，找到之前的commit ID，再reset回去

分支管理

基本操作

在某支线创建新支线，新支线会继承该支线的所有版本内容，且两条线处于同一版本A，此时对于其中一条线的改动会同步到另一条线上。但一旦有一条线commit了，处于不同版本B了，那么另一条线对于刚刚的改动全部撤销，仍然处于版本A，且由于处于不同版本，之后两条线的操作互不影响（产生分支）
简而言之，就是在支线a中做的一切操作都不会作用于支线b

查看，创建，删除，切换

合并分支
将指定分支的内容合并到当前分支（相当于把内容复制粘贴到当前分支），自动commit

解决冲突
手动去文件解决冲突后，git add添加到暂存区让git知道冲突已经解决，然后再commit上去

git diff—————-查看冲突的具体情况

变基分支

优选

对比merge和rebase

rebase机制：嫁接
每个分支都有一个指向当前分支最新提交记录的HEAD指针
rebase的机制是，先找到当前分支和目标分支的共同祖先，然后把当前分支的最新提交到共同祖先之间的所有提交记录都移动到目标分支

比如以上示例，两条分支共同的祖先是main:3
在dev分支上使用git rebase main，则会把main:3dev:2之间所有提交记录移到main分支上
在main分支上使用git rebase dev，则会把main:3main:5之间所有的提交里路移到dev分支上

如何选择

merge优缺点
优点：不会破坏原分支的提交历史，方便回溯和查看
缺点：产生额外的提交分支，分支图复杂

reabse优缺点
优点：不会新增额外的提交记录，形成线性，简洁直观
缺点：会改变提交历史，改变当前分支branch out的节点。应该避免在共享分支上使用

远程仓库

如果要想进行多人协作，就必须使用远程仓库。将本地仓库的数据同步到远程仓库，实现多人协作开发。
常见的Git远程仓库平台：GitHub，GitLab，Gitee

先添加一个远程仓库

再将文件推送到远程仓库
远程仓库和本地仓库一样，也有分支
如果本地分支名称和远端分支名称一致，那么不用指定远端分支名称，但是如果我们希望推送的分支和远端的分支不同名，那么需要额外指定

抓取，拉取
在多个本地仓库对应一个远程仓库的情况下，比如一个团队里面，N个人都在使用同一个远程仓库，但是他们各自只负责编写和推送自己业务部分的代码，也就是我们常说的协同工作，那么这个时候，我们就需要协调。

比如程序员A完成了他的模块，那么他就可以提交代码并推送到远程仓库，这时程序员B也要开始写代码了，由于远程仓库有其他程序员的提交记录，因此程序员B的本地仓库和远程仓库不一致，这时就需要有先进行pull操作，获取远程仓库中最新的提交

从远程仓库下载项目

多人开发

大概流程

  1. 克隆项目，将远程仓库的项目拉到本地
  2. 创建自己的开发分支a：不能在master分支进行代码编写，包括自己本地分支的master
  3. 在自己本地分支a中编码
  4. 自己本地分支a完成且无bug，则commit，然后切换到本地master分支，将自己分支代码a合并到master分支
  5. 本地master分支测试无bug后，将本地master分支推送到远程仓库master分支
  6. 每次编写前，先拉取远程仓库master最新代码到本地master分支
  7. 切换到自己本地分支a，合并master，然后继续开发

注意：自己本地的开发分支a也可以创一个远程仓库进行备份
